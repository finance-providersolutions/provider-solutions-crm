Phase 2 providers pages. /providers is a searchable list with status, specialty, and search filters, photo thumbnails per row, and a hide-archived toggle that defaults to on (so the imported ADMIN test row and Sample Provider row are hidden until the user explicitly looks for them). /providers/:id is the detail page with a header photo at xl-circle, contact info grid, activity feed, polymorphic log-activity form, placement and credentialing placeholder sections, and a destructive delete at the bottom. ProviderFormDialog handles create and edit with photo upload; the create flow generates the row's UUID up front so storage paths line up with the eventual insert.

Phase 2 opportunities pages. /opportunities is the demand-pipeline table with stage, specialty, state, and source-partner filters plus a search box. The source-partner filter dropdown lists "Any source", "Direct (no partner)", and one entry per partner organization. Each row shows the hospital name, opportunity title, specialty, location, stage badge, source partner (plain text "Medicus Healthcare Solutions" or "Direct"), and next-action date. /opportunities/:id is the detail page with header showing hospital name, optional "via [partner name]" plain-text rendering when a source partner is set, stage badge, and an Edit button. Below the header sit the overview grid, the full rate-structure grid (shift defaults, bill rates, pay rates, on-call when enabled, travel costs), the live GP modeler section, an activity feed and log-activity form, suggested-providers and tasks placeholder sections, and a destructive delete. OpportunityFormDialog is a sectioned create/edit dialog with searchable hospital and source-partner comboboxes (using the shadcn command primitive inside a popover) that include inline-create flows for "+ Create new hospital" and "+ Create new partner" — both seed type=hospital or type=locums_partner respectively into OrganizationFormDialog via its initialValues prop, so the user does not have to pick the type a second time.

Phase 2 GP modeler. The opportunity detail page hosts a live gross-profit projector. The pure compute() function and setting-aware seedDefaults() live in src/utils/gp-modeler.js so the formula stays unit-testable; the GPModeler component handles the form state, the Save and Reset buttons, and the projection grid. Defaults are setting-aware: inpatient opportunities seed with a 7-day shift, 7 working days, and on-call covering all 7 nights; outpatient opportunities seed with a 4-day shift, 4 working days, and no on-call. When setting is null or "other," the modeler falls back to the inpatient defaults. The user adjusts the assumption inputs in real time and the projection grid updates without a Calculate button. Estimates render in italic with a leading tilde and the warning amber color. Save persists the assumption blob (with all values coerced to numbers so the jsonb round-trips cleanly); Reset is a local-only revert to the setting-aware seed values without touching the database.

Phase 2 travel cost extension. supabase/migrations/0003_travel_costs.sql adds ps_covers_travel and three travel rate columns to opportunities. The OpportunityFormDialog has a Travel costs section that shows the three rate fields only when ps_covers_travel is checked, and clears them when the flag flips off — matching the on-call clear-on-disable pattern. The opportunity detail page renders a Travel costs subsection in the rate-structure grid that shows the three rate values when PS covers travel and a muted "Hospital covers travel" line otherwise. The GP modeler grew a Travel column in the projection grid: when PS covers travel, it shows the magnitude in parentheses with the italic-tilde-warning treatment as a deduction from GP; when the hospital covers travel, it shows "$0" in muted gray to signal that this is a fact on the opportunity record, not a projection. Compute extends to per-shift, weekly, monthly, and annual travel cost using the shift_length-aware default quantities (hotel_nights and rental_days default to shift_length + 1 for the arrival/departure travel buffer; airfare_trips defaults to 1). A mergeAssumptions helper in utils/gp-modeler.js layers the saved blob over setting-aware defaults so blobs persisted before travel keys existed don't render blank inputs after the schema grows; saved keys win over defaults, unknown keys pass through.

Phase 2 tasks page. /tasks renders three tabs ("My open" scoped to the current user, "All open" across the team, "Completed (30d)" filtered by completed_at) with a search-by-title input that filters within the active tab, a quick-complete checkbox on each open-task row that sets status='completed' and completed_at=now() in one update, click-the-title to open the edit dialog, and overdue-red styling on the due date when an open task is past today. TaskFormDialog has a parent picker that radio-toggles between organization, opportunity, provider, or no parent — three inline Command-in-Popover comboboxes backed by their respective resource hooks — and a left-aligned destructive Delete button in the footer (edit mode only) that hard-deletes the task after confirmation via the shared ConfirmDeleteDialog wrapper. The assignee picker is a temporary "Assign to me / Unassigned" toggle while we wait for the profiles table; the underlying assignee_id column is just a UUID, so the proper picker swaps in cleanly later. Each detail page (Opportunity, Provider, Organization) embeds a parent-scoped TasksSection so a task linked to Birmingham Grandview surfaces from Birmingham's detail page directly — same render shape as /tasks narrowed to one parent, with a "Show completed (30d)" toggle and a New Task button whose form locks the parent type and record so the user can't accidentally re-parent during the create flow. The full Phase 2 surface ships to Cloudflare via the provider-solutions-crm project on the next git push to the connected branch; the dashboard runs the build and `wrangler deploy` server-side and the production URL is confirmed there once the deploy completes.

## Shipped in CRM redesign implementation — Slice 3

Slice 3 — Opportunities list page swapped from the shadcn Table to a refined card-based list. After four design passes including a two-variant side-by-side comparison, the initial shape was one card per row at every width from phone through desktop, no two-up grid, no width-driven internal reflow. The source channel ("via Medicus" / "Direct") was removed from the card during iteration; it remains a filter dimension and continues to surface on the detail page. The kebab opens a dropdown menu with Edit (opens the existing OpportunityFormDialog in edit mode) and Delete (routes through the shared ConfirmDeleteDialog with cascading-delete warning); the menu auto-closes on selection. Whole-card tap navigates to the detail page; Enter and Space on the focused card do the same. The open-task-count source is a single useTasks fetch at the page level filtered by status open, bucketed into a per-opportunity map by useMemo so each card reads its count from local state — no per-card query, no N plus one. Count text renders in accent teal as "N tasks"; the numeral switches to danger red with medium weight when at least one linked open task has a due date strictly before today. Zero counts hide the line entirely. The Sort row in the filter sheet expanded from two options to five: Next action soonest (default, unchanged), Newest first (unchanged), and three new options — Hospital A to Z (sorts by joined organizations.name), Title A to Z, and Location state-then-city (state first with org fallback, then city with org fallback). All non-default sorts run client-side in the existing rows useMemo against the already-fetched dataset; no useOpportunities query changes were needed since the joined org table is already eagerly selected. Specialty abbreviation lives in src/utils/constants.js as a SPECIALTY_ABBR map and specialtyAbbrFor helper, with one entry today (Gastroenterology to Gastro). Filter sheet behavior, search, sort, row count footer, primary header, and two-tier list-page header are all unchanged. The contact CRUD gap noted in STATE.md known-issues is unchanged and remains Slice 4's natural fix venue.

A follow-on tune-up refined the card to a two-layout responsive system. Below the md breakpoint (768px) the card is a stacked block: 80x80 logo anchored on the left with the four-row meta-block vertically centered to its right; row 1 holds position with specialty and setting in white mono alongside the task count and kebab, row 2 holds the opportunity title in accent teal as the visual primary, row 3 holds the hospital name in white sans, row 4 holds city and state in a brighter muted-blue token (text-dim, swapped from the previous text-muted) alongside the stage badge. Row spacing is intentionally asymmetric — tight between rows 1 and 2 (the title sits visually flush with the position line via a negative top margin that compensates for DM Serif Display's intrinsic line-box padding), a larger break between rows 2 and 3 (title-to-hospital is the dominant rhythm point), tight between rows 3 and 4. At md and above the card switches to a single horizontal row: a smaller logo (56px bumping to 64px at lg) on the far left, two text clusters in the middle (left cluster stacks hospital name on top of city/state; center cluster stacks position-spec-setting on top of the title), and a right-anchored indicator cluster (task count plus kebab on top, stage badge below). The breakpoint switch is clean — no intermediate transitional state. Card padding tightened from p-4 to p-3 on the mobile layout so the logo sits closer to the card edges top and bottom, matching the left-side gap. The tasks count itself sits in a small outlined pill that mirrors the stage badge's visual language — digit only on mobile (the pill becomes a compact circle), "N tasks" at md and above where there is room for the word. The mobile position row uses tracking-tight letter-spacing on the mono font so "M.D. · Gastro · Inpatient" compresses horizontally, and the gap between the tasks pill and the kebab tightens to 4px on mobile (vs 8px on wide) so row 1 reads less crowded. All other behavior unchanged.

## Shipped in CRM redesign implementation — Slice 4

Slice 4 — the responsive card pattern extended to the four remaining list pages (Tasks, Contacts, Providers, Organizations), plus two minimal new detail pages and the resolution of the Contacts CRUD gap. Each page got its own audit-implement-iterate pass with per-page card decisions surfaced and signed off before implementation. The shared card grammar (logo slot, accent-teal primary identifier, mono dim secondary descriptor, lifecycle-state badge slot on the right, kebab with Edit + optional middle item + Delete) holds across all five pages; per-page variations exist where the data warrants and are documented in STATE.md's design-system section.

Tasks cards drop the logo slot entirely — tasks carry no identity image, and forcing a parent thumbnail across mixed parent types added chrome without payoff. The card uses up to 4 mobile rows with priority as a mono cap label (not a chip; the badge slot stays empty so the suite-wide grammar holds), the task title as the accent-teal primary, the parent identifier with mono cap type prefix, and a due-date plus status badge bottom row. Opportunity-parented tasks render an "at {hospital}" sub-line under the parent name because the opportunity title alone rarely identifies which hospital a task belongs to. The new /tasks/:id detail page mirrors Provider's shape with no thumb, a header with priority and status badges, a DetailGrid covering due / assignee / parent (linked) / notes, and a page-level destructive delete at the bottom. TaskFormDialog gained a hideDeleteAction prop so the new detail page's Edit dialog hides its in-dialog Delete (the page-level Delete is canonical there) while list-context callers leave the prop unset and keep the dialog Delete visible — every entry point to a task has exactly one delete affordance. useTask(id) is the new singular hook backing the detail page.

Contacts cards use the parent org's logo as the thumbnail (every contact has a required parent org, so the thumb is uniformly available). The card has a 4-row mobile pattern: contact name plus kebab on row 1, "ROLE at" lead-in on row 2 reading as a preposition flowing into row 3's organization name, and the title on row 4 — rows 2 and 4 hide gracefully when role or title is absent. Email and phone live on the new Contact detail page as mailto/tel links, not on the card; the line cluttered the card without aiding scanning. The new /contacts/:id detail page mirrors Provider's shape with no thumb, a DetailGrid covering title / role / email / phone / linked organization / notes, and a page-level destructive delete. ContactFormDialog has no in-dialog Delete (unlike TaskFormDialog), so no hideDeleteAction prop was needed. useContact(id) is the new singular hook.

The Contacts CRUD gap is resolved end-to-end. The Contacts list page now carries the always-visible kebab plus whole-card tap to the new detail page. The Organization detail page's embedded contacts list, which previously used hover-revealed Pencil and Trash2 icons (invisible on touch devices), now uses the shared CardKebab with always-visible Edit and Delete plus whole-row tap to the Contact detail page; mailto and tel anchors inside the row stop propagation so taps on them don't bubble.

Providers cards use the provider's own photo as the thumbnail (the strongest logo-slot case in the suite — a real per-record identity image with initials fallback). The card has a 3-row mobile pattern (position·specialty mono cap / provider name in accent teal / city·ST plus status badge) because providers are the leaf record and have no parent identifier to surface. The kebab carries a per-page deviation: Archive/Unarchive sits as a middle item between Edit and Delete because archiving is a frequent single-flag-flip lifecycle action that doesn't deserve the Edit-dialog round-trip. Archived providers render at opacity-50 on the whole card. The detail page already existed from Phase 2 and serves as the mirror source for the Tasks and Contacts detail pages built in this slice.

Organizations cards use the org's own logo as the thumbnail and a 3-row mobile pattern (type mono cap / name / city·ST). Type renders as a mono cap label, not as a chip in the badge slot — categorization, not lifecycle state, per the suite-wide grammar; the existing colored TYPE_BADGE palette continues to serve the detail page header where chip treatment is appropriate. The parent-scoped pill counts active opportunities at the org — lead, qualified, proposal, and contracted only; filled and lost are terminal and excluded via an explicit allowlist so a future new stage doesn't silently inflate the headline number. Pill copy spells "opportunity" and "opportunities" in full at md+ rather than abbreviating. Org names wrap to two lines (line-clamp-2) so the disambiguating "Medical Center" or "Health System" suffix isn't lost to single-line truncation — card height varies within the page as a consequence, accepted as the cost of preserving identity.

Two cross-cutting changes landed alongside the per-page work. The CardKebab component was extracted to live alongside the ConfirmDeleteDialog wrapper as a shared CRM-only abstraction; the trigger and dropdown structure that had been duplicated across Opportunities and Tasks is now a single source of truth, and an extraItems prop accepts lifecycle actions to render between the fixed Edit (first) and Delete (last) items. fmtName in utils/formatters.js was extended to join first + middle (rendered as-is, not forced to initial) + last + ", suffix" — middle and suffix render when the schema carries them (providers do, contacts don't), and the helper degrades gracefully so contacts still render correctly as "First Last". The card location line dropped one font-size step on Opportunities, Providers, and Contacts cards for visual calm. Archived opacity on Providers bumped from 60 to 50 for a stronger signal.

## Shipped in Phase 3 — slice 3a (credentialing core)

Three migrations landed together. 0004_credentialing.sql adds the layered credentialing tables — provider_licenses (state CHECK-constrained to the 50 US states + DC), credentials (provider-level core credentials with free-text label required when type='other' so multiple "other" rows stay distinguishable), and facility_privileges (per-hospital bridge with the full lifecycle status set plus the two terminal outcomes denied / withdrawn) — and the private credentials storage bucket with authenticated-only read/write policies for signed-URL document access. 0005_provider_status_values.sql swaps the providers.status CHECK constraint to the new 10-value list (target, lead, contacted, interested, interviewing, onboarding, active, inactive, declined, disqualified) with NOT VALID so existing 'credentialed' rows are grandfathered; the old 'disqualified' is split into declined (they walked) and disqualified (we screened them out). 0006_credentialing_application_dates.sql adds nullable application_date to provider_licenses and credentials so all three credentialing tables share one four-state computed status model.

Three credentialing sections on the provider detail page. State licenses, core credentials, and facility privileges each render as embedded add/edit/list sections under a single "Credentialing" SectionHeader, reading as one unit. Status is display-only — derived at render time from application/granting/expiration dates via deriveCredentialingStatus in src/components/credentialing/expiration.js; no manual status picker on any of the three form dialogs. Document uploads use the new DocumentUpload component (accepts PDF + JPG/PNG up to 10 MB, lazy-fetches a fresh signed URL on each "View" click). Facility privileges' kebab additionally carries Mark denied / Mark withdrawn / Clear outcome actions because those terminal outcomes can't be derived from dates and override the date-driven display.

The shared ExpirationCluster. Reference date pattern for detail pages — [amber countdown pill] + "Exp [date]" with rules in src/components/credentialing/ExpirationCluster.jsx: pill only shows for expirations within 90 days; status='expired' renders nothing (the red Expired badge carries it); missing date renders muted "No expiry". Used inline by all three credentialing section rows on the provider page and (with date-only mode) by the cross-provider dashboard.

Cross-provider expiration roll-up at /expirations. Read-only monitoring view combining all three credentialing tables across all providers; uses the suite's two-tier list-page chrome (bar-2 title + description + Search/Filter/Refresh icons, conditional bar-3 search input, slide-in filter sheet with Type + Sort rows). Items group into four bucket sections — Past expiration / Next 30 days / 31-60 days / 61-90 days — bucketed via the same expirationBucket helper that drives the card pills. Mobile rows show just the compact pill at row 2 right (no "Exp [date]" text — the bucket header carries urgency); desktop rows show the full inline cluster. Nav entry added with the CalendarClock icon.

Provider detail page reformat — the reference design for detail pages. Fixed condensed header below the primary PageHeader carrying photo + name (smaller on mobile, larger sm+, one-line), contact row (city/ST + email/phone icons), badge row (position type + specialty abbreviated via specialtyAbbrFor + provider pipeline status + archived), and an Edit button (icon-only mobile, icon + label sm+). Status badge sits right-edge under the Edit button, separated from the position/specialty group. Body section order: Details (collapsible, default collapsed — every editable field NOT in the header in a dense label-value grid, no card wrapper, one-off collapsible not a generic accordion), Credentialing cluster, Activity, Tasks, Placements placeholder, Delete footer. Activity uses a Provider-local "+ New activity" button that toggles LogActivityForm open/closed (not yet applied to Organization/Opportunity detail pages — they continue showing the always-visible form). No-card empty states applied to ActivityFeed (shared, affects Provider/Organization/Opportunity/Home) and TasksSection (shared, affects all three detail pages). A MutationObserver on document.body hides the fixed provider header whenever any role="dialog" with data-state="open" appears, so dialog tops clear both the PageHeader and the provider header.

Shared Dialog primitive change. src/components/ui/dialog.jsx is now top-anchored at calc(58px + safe-area + 1rem) instead of vertically centered, so dialog tops clear the 58px primary header on every page in the app. Affects every Dialog consumer (ProviderFormDialog, OrganizationFormDialog, ContactFormDialog, OpportunityFormDialog, TaskFormDialog, LicenseFormDialog, CredentialFormDialog, FacilityPrivilegeFormDialog) but not ConfirmDeleteDialog (alert-dialog primitive, unchanged). On list pages with bar-2 chrome at z-150, the dialog top can sit partially under bar-2 — known issue tracked in STATE.md and queued as next session's first item.

Direct ingestion of QBO actuals into CRM tables — parked. The financial dashboard remains the source of truth for actuals. The CRM will fetch via the qbo-proxy worker on demand instead of duplicating data.

Soft-delete with deleted_at — parked for Phase 1. Acceptable to revisit if accidental hard-deletes cause real data loss; today the cascade is gated only by a styled AlertDialog confirmation.

Provider name fallback to "Unknown" — parked for Phase 1/2. The Provider create/edit submit handler enforces "at least one of first or last name is filled" but builds the database payload with the literal string "Unknown" for whichever field is left blank, so a row can land with first_name "Bob" and last_name "Unknown" (or vice versa). Three viable fixes — allow null in the DB schema, require both fields filled in the UI validation, or build the payload with null instead of "Unknown" — and the right choice depends on how Phase 3 credentialing and Phase 4 matching will use these fields. Deferred until those usage shapes clarify.

TypeScript migration — parked. The dashboard is plain JSX and the CRM matches for suite consistency. Revisit only if a third suite app starts in TS.

React Query or SWR — parked. Custom hooks per resource are simple, mirror the dashboard, and are sufficient at current data volumes.

Shared component library across the suite — parked permanently. Each app owns its components. shadcn copies primitives into the repo by design. Consistency comes from shared tokens, not shared components.

Shared tokens.css extraction into a package — parked until a third suite app exists or token drift becomes painful. Today tokens are manually mirrored from the dashboard.

Light-mode token tuning — deferred to suite level. The CRM-redesign prototype surfaced that borders feel under-defined in light mode and that status-indicator contrast (income green, danger red, accent teal) is weaker than in dark mode. Tokens are mirrored CRM ↔ dashboard manually, so this is a cross-app concern and is owned at the suite level — the CRM is downstream of whatever happens there.
