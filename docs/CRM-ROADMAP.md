Phase 2 opportunities pages. /opportunities is the demand-pipeline table with stage, specialty, state, and source-partner filters plus a search box. The source-partner filter dropdown lists "Any source", "Direct (no partner)", and one entry per partner organization. Each row shows the hospital name, opportunity title, specialty, location, stage badge, source partner (plain text "Medicus Healthcare Solutions" or "Direct"), and next-action date. /opportunities/:id is the detail page with header showing hospital name, optional "via [partner name]" plain-text rendering when a source partner is set, stage badge, and an Edit button. OpportunityFormDialog is a sectioned create/edit dialog with searchable hospital and source-partner comboboxes (using the shadcn command primitive inside a popover) that include inline-create flows for "+ Create new hospital" and "+ Create new partner" — both seed type=hospital or type=locums_partner respectively into OrganizationFormDialog via its initialValues prop, so the user does not have to pick the type a second time.

Phase 2 GP modeler. The opportunity detail page hosts a live gross-profit projector. The pure compute() function and setting-aware seedDefaults() live in src/utils/gp-modeler.js so the formula stays unit-testable; the GPModeler component handles the form state, the Save and Reset buttons, and the projection grid. Defaults are setting-aware: inpatient opportunities seed with a 7-day shift, 7 working days, and on-call covering all 7 nights; outpatient opportunities seed with a 4-day shift, 4 working days, and no on-call. When setting is null or "other," the modeler falls back to the inpatient defaults. The user adjusts the assumption inputs in real time and the projection grid updates without a Calculate button. Estimates render in italic with a leading tilde and the warning amber color. Save persists the assumption blob (with all values coerced to numbers so the jsonb round-trips cleanly); Reset is a local-only revert to the setting-aware seed values without touching the database.

Phase 2 travel cost extension. supabase/migrations/0003_travel_costs.sql adds ps_covers_travel and three travel rate columns to opportunities. The OpportunityFormDialog has a Travel costs section that shows the three rate fields only when ps_covers_travel is checked, and clears them when the flag flips off — matching the on-call clear-on-disable pattern. The opportunity detail page renders a Travel costs subsection in the rate-structure grid that shows the three rate values when PS covers travel and a muted "Hospital covers travel" line otherwise. The GP modeler grew a Travel column in the projection grid: when PS covers travel, it shows the magnitude in parentheses with the italic-tilde-warning treatment as a deduction from GP; when the hospital covers travel, it shows "$0" in muted gray to signal that this is a fact on the opportunity record, not a projection. Compute extends to per-shift, weekly, monthly, and annual travel cost using the shift_length-aware default quantities (hotel_nights and rental_days default to shift_length + 1 for the arrival/departure travel buffer; airfare_trips defaults to 1). A mergeAssumptions helper in utils/gp-modeler.js layers the saved blob over setting-aware defaults so blobs persisted before travel keys existed don't render blank inputs after the schema grows; saved keys win over defaults, unknown keys pass through.

Phase 2 tasks page. /tasks renders three tabs ("My open" scoped to the current user, "All open" across the team, "Completed (30d)" filtered by completed_at) with a search-by-title input that filters within the active tab, a quick-complete checkbox on each open-task row that sets status='completed' and completed_at=now() in one update, click-the-title to open the edit dialog, and overdue-red styling on the due date when an open task is past today. TaskFormDialog has a parent picker that radio-toggles between organization, opportunity, provider, or no parent — three inline Command-in-Popover comboboxes backed by their respective resource hooks — and a left-aligned destructive Delete button in the footer (edit mode only) that hard-deletes the task after confirmation via the shared ConfirmDeleteDialog wrapper. The assignee picker is a temporary "Assign to me / Unassigned" toggle while we wait for the profiles table; the underlying assignee_id column is just a UUID, so the proper picker swaps in cleanly later. Each detail page (Opportunity, Provider, Organization) embeds a parent-scoped TasksSection so a task linked to Birmingham Grandview surfaces from Birmingham's detail page directly — same render shape as /tasks narrowed to one parent, with a "Show completed (30d)" toggle and a New Task button whose form locks the parent type and record so the user can't accidentally re-parent during the create flow. The full Phase 2 surface ships to Cloudflare via the provider-solutions-crm project on the next git push to the connected branch; the dashboard runs the build and `wrangler deploy` server-side and the production URL is confirmed there once the deploy completes.

After four design passes including a two-variant side-by-side comparison, the initial shape was one card per row at every width from phone through desktop, no two-up grid, no width-driven internal reflow. The source channel ("via Medicus" / "Direct") was removed from the card during iteration; it remains a filter dimension and continues to surface on the detail page. Contacts cards use the parent org's logo as the thumbnail (every contact has a required parent org, so the thumb is uniformly available). The card has a 4-row mobile pattern: contact name plus kebab on row 1, "ROLE at" lead-in on row 2 reading as a preposition flowing into row 3's organization name, and the title on row 4 — rows 2 and 4 hide gracefully when role or title is absent. Email and phone live on the new Contact detail page as mailto/tel links, not on the card; the line cluttered the card without aiding scanning. The Contacts CRUD gap is resolved end-to-end. The Contacts list page now carries the always-visible kebab plus whole-card tap to the new detail page. The Organization detail page's embedded contacts list, which previously used hover-revealed Pencil and Trash2 icons (invisible on touch devices), now uses the shared CardKebab with always-visible Edit and Delete plus whole-row tap to the Contact detail page; mailto and tel anchors inside the row stop propagation so taps on them don't bubble.

Providers cards use the provider's own photo as the thumbnail (the strongest logo-slot case in the suite — a real per-record identity image with initials fallback). The kebab carries a per-page deviation: Archive/Unarchive sits as a middle item between Edit and Delete because archiving is a frequent single-flag-flip lifecycle action that doesn't deserve the Edit-dialog round-trip. Organizations cards use the org's own logo as the thumbnail and a 3-row mobile pattern (type mono cap / name / city·ST). Org names wrap to two lines (line-clamp-2) so the disambiguating "Medical Center" or "Health System" suffix isn't lost to single-line truncation — card height varies within the page as a consequence, accepted as the cost of preserving identity.

The card location line dropped one font-size step on Opportunities, Providers, and Contacts cards for visual calm. Archived opacity on Providers bumped from 60 to 50 for a stronger signal.

## Shipped in Phase 3 — slice 3b (readiness, requirements, onboarding, restructure)

A pure zero-fetch shift-readiness engine — deriveShiftReadiness in src/components/credentialing/readiness.js. Scores a single provider against an opportunity using already-loaded credentialing data; reuses deriveCredentialingStatus from expiration.js without reimplementing date logic. Returns a structured result with an overall portable-only roll-up of license plus core credentials, a facility companion that is reported but never folded into overall (so an unplaced candidate without a privilege row at the hospital does not read as uniformly blocked), the resolved licensure state, and a flat machine-readable reasons array with named codes for each failure mode. Requirements gate every dimension — a dimension not in the opportunity's required_items is skipped, not blocked. Roll-up precedence is blocked > indeterminate > expiring > ready; an expiring required component flips overall to expiring. The same helper will be the credential half of Phase 4 matching — it should be consumed there, not rebuilt.

Opportunity requirements as data. 0007_opportunity_requirements.sql adds a nullable required_items text[] column to opportunities — no default, no CHECK; null and empty read identically as "not assessable yet." REQUIREMENT_ITEMS in constants.js is the seven-key master list (state license, the five real credential types sourced from CREDENTIAL_TYPES minus `other`, and facility privileges) — values are byte-identical to the keys the readiness helper checks. OpportunityFormDialog gained a Requirements section that toggles which items an opportunity requires and writes the selected keys back into required_items. Opportunity detail shows a compact readout near the Suggested-providers placeholder — "Requires: …" using REQUIREMENT_ITEMS labels, or muted "Requirements not defined" when unset. useOpportunities' hospital join was widened to include organization.state and city so the readiness helper can resolve licensure state without a second fetch; the source-partner join is unchanged.

Catalog-driven provider onboarding. 0008_onboarding.sql adds onboarding_item_types (the catalog — seeded with cv, references repeatable, and background_check) and onboarding_items (per-provider rows with a done toggle, completed_date auto-stamped on toggle-on, and an optional document_path into the existing private credentials bucket from 0004 — no new bucket). The item_key column on onboarding_items is a real FK into the catalog with `on delete restrict` — the first editable-catalog FK in the app and the template for future admin-managed list tables. New hook useOnboarding.js sibling to useCredentialing and useExpirations. An admin UI for adding new catalog item types is intentionally deferred — the catalog ships seeded and is extended by manual insert until the polish-pass admin surface lands.

Provider detail page restructure. Section order is now Header → Details → Onboarding → Credentialing → Activity → Tasks → Placements → Delete. Onboarding moved above Credentialing to mirror the onboard-then-credential lifecycle. Both Onboarding and Credentialing now follow ONE pattern: a diamond SectionHeader, an always-visible centered status indicator, and a default-collapsed body behind a shared in-house CollapsibleSection (added at src/components/ui/collapsible-section.jsx — mirrors the existing Details one-off's mechanics, generalized to label + children). Details itself was intentionally left as its own one-off and was not migrated onto the new component. CredentialingSection.jsx is a new wrapper that owns the worst-status-per-group summary and renders the three unchanged subsections (Licenses, Credentials, Privileges) inside the collapsible. Onboarding's status indicator is "N of 3 complete" + thin progress bar (denominator covers CV, references-satisfied-if-any-row-done, and background check; license/DEA are not counted). Credentialing's status indicator is a centered three-line worst-status-per-group summary using deriveCredentialingStatus and expirationBucket — count reflects rows in the worst bucket only, empty groups read "none on file," tone colors mirror the credentialing rows themselves. The previously-shipped derived "License on file" and "DEA on file" rows on the Onboarding section were removed in the same pass; their status now lives solely in the credentialing summary, so it isn't shown twice on one screen. The retired onboardingDerived.js helper was deleted.

5. Opportunity segment-type restructure (planned — demand-side mirror of the Scheduler's segment model). The Scheduler now models work as day/night SEGMENTS, each separately priced and separately signed (see ps-app-scheduler/docs/SCHEDULER-SCHEMA.md). The CRM Opportunity defines the base rates those segments inherit but still uses the old fused shape — a single rate form (today's RateStructureFormDialog) bundling regular-shift and night-shift start/end times and all bill/pay rates together — which no longer matches how the Scheduler consumes it. The restructure: an Opportunity is created in one form (the existing OpportunityFormDialog — Basics / Location / Dates / Pipeline), then a SEPARATE section adds SEGMENT TYPES, each carrying its own base bill and pay rates. Case distinction: an opportunity with no night call (e.g. the Billings Clinic GI Outpatient Clinic opp) has ONE segment type — a day shift with its rates; an opportunity with night call (e.g. the Billings Clinic 24hr GI Consult Coverage opp) has TWO — a day shift and a separate night shift, each with its own rates. The segment types do double duty: they establish demand-side base rates AND define which segment kinds the Scheduler is allowed to instantiate for the opportunity. In the Scheduler, scheduled segments inherit their type's base rates but can be adjusted per-instance where appropriate (holiday pay, advanced-shift, one-off negotiated rates) — the segment stores the actual figure that applied, the segment type is the template/default. Boundary to hold: this is a DEMAND-SIDE restructure that keeps the CRM's established role — segment TYPES with base rates are still "typical economics for negotiation," just reshaped to match how the work decomposes. It does NOT move scheduling or per-instance variance into the CRM; day-level per-instance reality stays a Scheduler concern. Coupled to the Scheduler's historical-data import: imported segments inherit base rates from these segment types, so the types must exist (or be derivable) before/with the import — sequence the two together. Cross-app; tracked at suite level in ps-apps-suite/docs/SUITE-ROADMAP.md and on the Scheduler side in ps-app-scheduler/docs/SCHEDULER-ROADMAP.md.
