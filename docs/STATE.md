# STATE — Provider Solutions CRM

A snapshot of where this app is right now. Scope is the CRM only. Cross-app concerns live in the suite-level docs at ps-apps-suite/docs/.

## What this app is

Internal CRM for Provider Solutions, LLC (LOCUMs physician staffing). Second app in a suite that already includes a live financial dashboard. Targets demand (opportunities from hospitals and LOCUMs partners) and supply (recruiting, credentialing, and managing providers). Real auth from day one. Users at launch are Jason and Reed; designed to scale to admin staff within months.

## Current phase

Phases 1 and 2 are shipped end-to-end. The full CRM redesign implementation arc (four slices covering primary-header chrome, two-tier list-page chrome, the Opportunities card swap, and the card swap on the four remaining list pages plus two minimal new detail pages) shipped on top. The thin Supabase-only half of the originally-planned pipeline-aware Home rebuild is now shipped — Home shows pipeline-aware KPIs (open opportunities by stage, active providers, tasks needing attention), the existing recent-activity feed, and a navigation hub. The later Home-enrichment pass with optional QBO actuals is still queued — its inputs (expiring credentials, stalled providers, uncredentialed opportunities) come from Phase 3b/4 and the QBO-Worker integration is no longer bundled into Phase 2.

Phase 3 — credentialing — is now mostly shipped. Slices 3a and 3b are both live; only 3c (the daily credential-alerts Supabase Edge Function via Resend) remains. 3a delivered the credentialing core — schema for `provider_licenses`, `credentials`, and `facility_privileges`; the private `credentials` storage bucket with signed-URL access; manual credential entry on the provider detail page; a cross-provider expiration roll-up at `/expirations`; computed credential status derived from dates at render time; and the provider detail page as the reference design for detail-page layout. 3b delivered three things on top of that: a pure zero-fetch shift-readiness engine (deriveShiftReadiness) that evaluates a provider against an opportunity's declared requirements; opportunity requirements as actual data (a flat required_items text[] column, a Requirements toggle picker in the opportunity form, a compact readout on opportunity detail); and a catalog-driven provider onboarding checklist with optional document attachments into the existing `credentials` bucket. 3b also restructured the provider detail page — Onboarding moved above Credentialing (matching the onboard-then-credential lifecycle), both sections gained an always-visible centered status indicator, and their bodies sit behind a shared default-collapsed CollapsibleSection. That status-indicator-plus-collapsible-body pattern is now the reference shape the polish pass should extend to the other detail pages. Phase 4 (matching and placement) follows Phase 3.

A read-only recruiting-funnel view at `/funnel` is also shipped, on top of the new 10-stage provider status. It groups providers by stage so the supply pipeline reads at a glance, with no schema change — purely a new presentation of data already in the providers table.

## Hosting and infrastructure

The app is hosted on Cloudflare as a static-assets deployment via the provider-solutions-crm project. The deploy config lives in wrangler.jsonc — the build runs vite, the dist/ folder is uploaded as assets, and Cloudflare's single-page-application not-found-handling serves index.html for any unmatched path so client-side routing works. The legacy public/_redirects file was removed once the wrangler config took over. Deploys are git-push-triggered: pushing to the connected branch causes the Cloudflare dashboard to run npm run build then npx wrangler deploy server-side. No local Cloudflare auth is needed; the production URL is confirmed against the Cloudflare dashboard after each deploy completes.

Backend is Supabase (single project, all environments share it for now). Supabase project ref is ztbadmaufcpkinnjztxy; the URL and publishable key are hardcoded in src/api/supabase.js, which is safe because Row Level Security is enabled on every table. The service role key never ships to the browser and lives only in the Supabase dashboard or in Jason's terminal env when running scripts.

The app is PWA-capable. index.html declares the manifest and Apple home-screen meta tags; public/manifest.webmanifest sets standalone display, dark theme color #0b1c2e, and uses pslogo.png as the icon. iOS safe-area-inset-top is respected on the fixed page header and on every page's top padding so the app sits below the status bar when installed.

When the CRM needs financial-dashboard data (QBO actuals against opportunities), the plan is to call the existing worker at qbo-proxy.finance-providersolutions.workers.dev rather than duplicating that data into Supabase. No CRM code calls that worker yet. That integration is now optional and deferred to a later Home-enrichment pass — gated on whether seeing opportunity-vs-actuals would change a weekly decision — and is no longer bundled into Phase 2.

Credentialing was originally penciled as 0003 but slipped to 0004 as a result. supabase/migrations/0004_credentialing.sql (Phase 3, slice 3a) adds three layered credentialing tables — provider_licenses (state medical licenses, provider-level, state CHECK-constrained to the 50 US states + DC), credentials (provider-level core credentials — board cert, DEA, BLS, ACLS, malpractice, other — with a free-text `label` required when type='other' so multiple "other" rows are distinguishable), and facility_privileges (per-hospital bridge with the full lifecycle status set: pending, active, expired, denied, withdrawn) — plus the private `credentials` storage bucket with authenticated-only SELECT/INSERT/UPDATE/DELETE policies (no anon read) so documents are served via createSignedUrl. supabase/migrations/0005_provider_status_values.sql swaps the providers.status CHECK constraint to the new 10-value list (target, lead, contacted, interested, interviewing, onboarding, active, inactive, declined, disqualified) with NOT VALID so existing 'credentialed' rows are grandfathered — those rows must be reassigned manually on next edit; a future VALIDATE CONSTRAINT pass can formally validate the historical data once all stale rows are gone. supabase/migrations/0006_credentialing_application_dates.sql adds a nullable application_date date column to provider_licenses and credentials so all three credentialing tables share the same four-state computed status model (applied → active → expiring → expired). supabase/migrations/0007_opportunity_requirements.sql (Phase 3, slice 3b) adds a nullable required_items text[] column to opportunities — no default, no CHECK; the column drives the readiness helper, which treats null and empty array identically as "not assessable yet" (REQUIREMENTS_UNDEFINED). The trust model matches modeling_assumptions — the Requirements picker is the only writer and only writes valid keys drawn from REQUIREMENT_ITEMS in constants.js, so a CHECK was deliberately omitted. supabase/migrations/0008_onboarding.sql (Phase 3, slice 3b) adds the two onboarding tables — onboarding_item_types (the catalog, seeded with `cv`, `references` repeatable, and `background_check`) and onboarding_items (per-provider persisted rows with done, completed_date, and an optional document_path into the existing private credentials bucket from 0004 — no new bucket). onboarding_items.item_key is a real FK into onboarding_item_types(key) with `on delete restrict`; this is the first editable-catalog FK in the app and is the template for future admin-managed list tables. Migrations are immutable once applied — never edit any of them again, ever; new changes require new numbered files.

## Data model and integrations

Twelve tables once 0008 is applied. organizations carries name, type (hospital, locums_partner, or other), website, address fields, notes, and now image / recruiting columns (logo_path, image_path, tourist_site_url, long_description) plus appsheet_id. The hospital-flavored image and recruiting columns remain on the table for every row but are only surfaced in the create/edit form when type is hospital or other; partner orgs only need name, website, address, contacts, and notes. contacts hangs off organization_id with name, title, role, email, phone, notes. providers carries name parts, contact info, NPI, specialty, position_type, home city/state, photo_path, travel-booking conveniences, status (the 10-stage recruiting pipeline updated by 0005), source, an archived flag, and appsheet_id. opportunities hangs off organization_id (the hospital) with an optional source_partner_id (the LOCUMs partner organization, e.g., Medicus — a relationship AppSheet never modeled, so legacy data is seeded by hand during import), title, name, position_type, specialty, setting, location, dates, the full bill-side and pay-side rate structure, an on-call window, a ps_covers_travel flag plus three travel-rate columns added in 0003, a jsonb modeling_assumptions column for the GP modeler, a required_items text[] added in 0007 (the flat list of credentialing items deriveShiftReadiness gates on — null/empty reads as "not assessable yet," not "ready by default"), pipeline stage and probability, next_action_date, notes, and appsheet_id. tasks carries title, due_date, status, priority, an assignee, and optional links to organization, opportunity, or provider. placements bridges provider_id and opportunity_id with start/end dates, status, and per-placement pay_rate / bill_rate. activities is the polymorphic touch log; with 0002 applied, all four parent FK columns now carry REFERENCES, and the Phase 1 CHECK enforcing exactly one parent is unchanged. The slice-3a credentialing tables are layered: provider_licenses (state medical licenses, the matching read path), credentials (provider-level core credentials with a free-text label required when type='other'), and facility_privileges (per-hospital bridge, the per-placement gate). All three share the same four-state computed status model derived from dates at render time (expired → active → applied → pending); the DB status column is kept in sync on writes but display never reads it for the lifecycle states. Facility privileges also carry two terminal outcomes (denied, withdrawn) that the kebab can set explicitly — terminal stored statuses override the date-driven display so a denied privilege never renders as Pending or Active. The slice-3b onboarding tables are layered too: onboarding_item_types is the catalog (key, label, repeatable boolean, sort_order — seeded with cv / references / background_check; no admin UI yet, new types go in by manual insert) and onboarding_items is the per-provider persistence layer (provider_id cascade, item_key FK into the catalog with on delete restrict, done boolean with completed_date auto-stamped on toggle-on and cleared on toggle-off, optional document_path into the existing credentials bucket).

Audit columns (created_at, updated_at, created_by referencing auth.users) appear on every user-editable table. UUIDs everywhere. No soft-delete — hard delete with cascade. CHECK constraints validate enum-like text columns and rate non-negativity; an opportunities.on_call_enabled flag gates the on-call rate columns through a CHECK requiring rates to be NOT NULL when enabled, while the 0003 ps_covers_travel flag gates the three travel rate columns through the inverse asymmetric CHECK requiring rates to be NULL when disabled.

External integrations today: Supabase auth and Postgres, plus three storage buckets — organization-logos and provider-photos (public-read, created in 0002, image rendering uses supabase.storage.getPublicUrl so URLs need no signing) and credentials (private, created in 0004, documents served via supabase.storage.createSignedUrl with a 5-minute expiry, authenticated SELECT/INSERT/UPDATE/DELETE only). The QBO proxy at qbo-proxy.finance-providersolutions.workers.dev is still the planned cross-app data source for opportunity actuals but is not wired up yet — that integration is optional and deferred to the later Home-enrichment pass, no longer bundled into Phase 2. AppSheet remains the legacy system providers use for shift logs; the appsheet_id columns now exist on providers, organizations, and opportunities for stable matching when the import script runs. There is no two-way sync, will never be one — AppSheet is being retired, not synchronized.

The Phase 3 credential-alerts Edge Function that emails a daily expiration digest to all.provider.solutions@gmail.com via Resend is still pending — that ships in slice 3c. Slice 3a manual entry plus the cross-provider expiration roll-up at /expirations have shipped.

## Auth and core domain rules

Auth is email OTP — Supabase emails a 6-to-10 digit code (length is configurable in the Supabase dashboard; the input accepts the full range so a project-side change does not lock anyone out). The login flow used to be magic link; recent commits switched it to OTP because the code flow stays inside the installed PWA without bouncing through Mail.

RLS Phase 1 strategy is simple and intentional: any authenticated user can SELECT, INSERT, UPDATE, and DELETE everything. Anonymous access is blocked. Phase 2-plus will introduce a profiles table and role-based policies (admin, recruiter, viewer) when admin staff are added.

Once Phase 2 ships, every new provider, organization, and opportunity goes into the CRM exclusively — AppSheet is frozen for those record types. AppSheet still receives shift logs from providers until the future provider portal replaces that workflow.

Migrations are immutable once shipped. Errors must surface via sonner toast with the full error logged to console — never silently caught. Money is rendered with Intl currency formatting; estimates are italic with a leading tilde and the warning color. localStorage keys are prefixed ps-crm-. Routes are one per page; detail pages take their parent id as a URL param. Plain JSX, no TypeScript.

Destructive actions go through the shared ConfirmDeleteDialog wrapper, which restores focus to the trigger element on cancel rather than dropping it to body. Long-form dialogs (ProviderFormDialog, OpportunityFormDialog) use a sticky save bar on phone and pair toast.error with scrollToFirstError to surface validation failures. Required-field labels carry an inline red asterisk via the Field component's required prop whenever a field triggers a validation error in the submit handler.

## Known issues and latent risks

Phase 1 RLS is permissive by design — any authenticated user can delete any record. Acceptable for two users, not acceptable once admin staff arrive. Add profiles + role policies before granting accounts beyond the founders.

Provider rows with the legacy 'credentialed' status value (from before 0005 applied) are grandfathered by 0005's NOT VALID constraint. They render with the fallback badge style via STATUS_BADGE_FALLBACK in src/pages/Providers.jsx. Editing such a row will fail the new CHECK unless the same edit also moves status to a value in the new 10-stage list — that failure surfaces via the existing toast.error path and is the prompt to reassign the status. Once all stale rows are gone, ALTER TABLE public.providers VALIDATE CONSTRAINT providers_status_check can be run manually as a someday cleanup.

Organization deletion cascades to contacts and activities through the schema, gated by a styled AlertDialog confirmation in the UI. There is no soft-delete and no undo. A misclick on a populated organization is destructive.

Activity type sms is in the schema and ACTIVITY_TYPES list but has no UI optimizations beyond the icon. LogActivityForm is now polymorphic and attaches cleanly to organizations, providers, and opportunities; the remaining gap is contacts — there is still no way to log an activity against a contact directly. Slice 4 added a Contact detail page, but the activity-against-contact link requires a schema change (activities currently has no contact_id link to a real contact record beyond the Phase 1 placeholder), so the gap persists. The minimal detail page does not try to fix it.

The Login.jsx OTP code step has a length minimum of 6 digits; if Supabase project settings ever drop below 6 the form will reject all valid codes. The pattern is set wide (6 to 10 digits) to absorb the documented configurable range, but a 4-digit setting would silently break login.

The Tasks page assignee picker is a simple "Assign to me / Unassigned" toggle rather than a real team-member picker. The schema's tasks.assignee_id references auth.users, which the Supabase JS client cannot enumerate without the service role key. The proper picker waits for the Phase 2-plus profiles table; until then, all tasks created in the UI are either assigned to the current user or left unassigned. Tasks created with another user's UUID via the API or migrated from elsewhere will still surface correctly in the "All open" tab, just without a name attached.

The Provider create/edit submit handler enforces "at least one of first or last name is filled" but falls back to the literal string "Unknown" for whichever field is left blank when building the database payload. A row can therefore land with first_name "Bob" and last_name "Unknown" (or vice versa). Three viable fixes — allow null in the DB schema, require both fields filled in the UI validation, or build the payload with null instead of "Unknown" — and the right choice depends on how Phase 3 credentialing and Phase 4 matching will use these fields. Parked product decision.

Token drift between this CRM and ps-app-dashboard is a manual sync today. tokens.css here mirrors the dashboard's index.css by hand. When a third app joins the suite, tokens extract to a shared package; until then, dashboard token edits must be ported to CRM by hand or visual divergence creeps in.

The detail-page top padding hard-codes the header height as 58px plus safe-area-inset-top in inline styles. If the header height changes, every page needs an edit. A CSS variable would centralize this.

The new Onboarding and Credentialing sections on the provider detail page default to collapsed. That trades upfront density for a calmer scan, but it adds one tap to the workflow of arriving from the /expirations roll-up to act on a specific credential — the link lands on the provider page with the Credentialing body still closed, so the user has to expand "Provider Credentials" before the row they came for is visible. A deep-link mechanism that opens the right collapsible (and ideally scrolls to the specific row) is queued for the polish pass, not fixed now.

## Design system at a glance

Dark default. Background #0b1c2e (deep navy), surface #122540 raised one level, surface2 #1a3050 raised two levels. Accent is teal #7ee8e8 with a brighter hover variant #00e0ff and a near-transparent fill accent-dim. Income green #3ecf8e for active, current, won states. Danger red #ff8080 for lost, expired, COGS. Warning amber #c8a840 is CRM-only — used for estimates (italic with tilde prefix) and soon-expiring credentials in Phase 3. Light theme is a separate :root block under data-theme="light"; ThemeToggle persists the choice to localStorage.

Three fonts. DM Serif Display for h1 and h2 only (font-display utility). DM Sans for body (font-sans, default). DM Mono for numbers, IDs, and small-caps labels (font-mono). Mono uppercase with wide letter-spacing is the recurring label and metadata treatment throughout.

Radius is 5px globally via the --radius variable.

Header icon buttons are 36×36, slightly smaller than the dialog close X at 40×40 — a context-specific choice justified by the dedicated chrome strip where no adjacent interactive elements compete for thumb taps.

List pages render a two-tier header. Below the primary header sits a second fixed bar carrying the page title and a sm+ description on the left and a three-icon action group on the right — Search, Filter, Plus — at the same 36×36 chrome-strip sizing. A "Clear" mono link appears in the second tier when any filter or search is active. A conditional third bar slides down below the second tier when search is toggled, hosting the search input. The filter sheet slides in from the right via the same Radix-style portal pattern as the Navigation drawer and contains per-page filters plus a sort row. List pages compute body padding dynamically — 58 + 56 + (52 if search open) + safe-area-inset-top — distinct from the 58px-only hardcode that non-list pages still use; that hardcode is unchanged and remains a known-issue item to centralize someday.

Shadcn-style overlays (Dialog, AlertDialog, the list-page filter sheet) all use z-50 and rely on DOM-order tiebreak for stacking among themselves so that Radix-portaled popovers like Select dropdowns render above the overlay that triggered them; chrome bars at z-150 (Bar 2 / Bar 3) and z-200 (the primary header) sit above overlays by design so the user can still orient on what page they are on while an overlay is open. The shared Dialog primitive anchors its top below all fixed chrome by reading a `--ps-chrome-bottom` CSS variable that pages publish via the useChromeBottom hook in src/hooks/useChromeBottom.js — list pages and the /expirations page set it to primary header + bar-2 + optional bar-3 (tracking searchOpen live so the variable updates when bar-3 slides in), the provider detail page sets it to primary header + its measured condensed header, and pages without page-owned chrome leave the variable unset so the dialog falls back to anchoring below the 58px primary header alone. Dialogs open below chrome, not under it; chrome stays visible during open.

All five list pages render rows as cards rather than as shadcn Tables. The card pattern is one card per row at every width with no two-up grid, but the card itself has two distinct layouts joined at the md breakpoint (768px). Below md, the card is a stacked block: a logo-slot thumbnail on the left and a meta block vertically centered to its right. At md and above, the card switches to a single horizontal row: a smaller thumbnail on the far left, content cluster (or clusters) in the middle, and a right-anchored indicator cluster. The breakpoint switch is clean — no intermediate transitional state, fully one layout below md and fully the other above. Whole-card tap navigates to the detail page; Enter and Space on the focused card do the same. The kebab opens a dropdown with Edit (opens the form dialog in edit mode), an optional middle item for lifecycle actions like Archive, and Delete (routes through the shared ConfirmDeleteDialog with cascading-delete warning when applicable); the menu auto-closes on selection. The kebab itself lives in the shared CardKebab component so all five pages stay in sync.

The card grammar is consistent across pages: the logo slot is the resource's strongest available image; the primary identifier renders in accent teal font-display; secondary descriptors render in white sans or mono dim; the badge slot on the right is reserved for lifecycle state (stage, status) and stays empty for resources without one. Per-page variations exist where the data warrants. Opportunities cards have a 4-row mobile pattern (position·spec·setting / title / hospital / city·ST + stage badge) with the org logo as the thumbnail. Tasks cards omit the logo slot entirely because tasks carry no identity image; the meta block claims full card width, the row 1 mono line carries priority as a cap label (with high priority in text-warning) alongside the quick-complete checkbox and kebab, the task title is the accent-teal primary on row 2, the parent identifier with its mono cap type prefix sits on row 3, and opportunity-parented tasks add an "at {hospital}" sub-line beneath the opportunity title because the title alone rarely identifies which hospital a task belongs to. Contacts cards use the parent org's logo (since every contact has a required parent org) and a 4-row mobile pattern with the contact name as the primary, a "ROLE at" lead-in below the name that reads as a preposition flowing into row 3's organization name, and a title row at the bottom — rows 2 and 4 hide gracefully when role or title is absent; email and phone live on the detail page as mailto/tel links, not on the card. Providers cards use the provider's own photo (the strongest logo-slot case in the suite) and a 3-row mobile pattern (position·spec / name / city·ST + status badge); the kebab carries Archive/Unarchive as a middle item between Edit and Delete, and an archived provider renders at opacity-50 on the whole card. Organizations cards use the org's own logo, a 3-row mobile pattern (type mono cap / name / city·ST), and a parent-scoped pill counting active opportunities only — lead, qualified, proposal, and contracted; filled and lost are terminal and excluded. Org names systematically need to wrap to two lines because the disambiguating "Medical Center" or "Health System" suffix is exactly what single-line truncate cuts; card height varies within the Organizations page as a consequence, which the design accepts.

Type, role, and similar categorization signals render as mono cap labels rather than as colored chips in the badge slot, even when a colored palette already exists for them — the badge slot stays semantically pure as the lifecycle-state slot across the suite. The colored TYPE_BADGE palette stays alive on the Organization detail page header where chip treatment is appropriate for a structured detail surface.

The parent-scoped pill on Opportunities, Providers, and Organizations cards follows one shared shape: a single page-level fetch of the relevant resource (open tasks for Opportunities and Providers, opportunities for Organizations) gets bucketed into a per-parent map by useMemo so each card reads its count from local state — no per-card query and no N plus one. The pill is a small outlined chip. The numeral renders in accent teal normally; on the task pills it switches to danger red with medium weight when at least one linked open task is overdue. Zero counts hide the pill entirely. The Organizations opportunity pill spells the unit word ("opportunity" / "opportunities") in full at md+ rather than abbreviating. The Provider-card task pill carries the unit word ("task" / "tasks") at every width now, including mobile — the overlap fix below and the status-badge suppression on the funnel freed enough row-1 space that the digit alone was unnecessarily terse, and the wider pill reads better in both contexts.

ProviderCard is a single shared component used by both the Providers list and the Funnel, parameterized by a showStatus prop and by the presence or absence of edit/archive/delete handlers — the kebab disappears entirely in read/navigate-only contexts (the funnel). A latent vertical-rhythm bug was found and fixed during the funnel build: a `-mt-1` on the H3 name row was pulling the line-box into the position·spec row above, overlapping by 4px on every card that lacked a tall row-1 right-side element. Removed; the card now uses a small positive gap between rows instead. The bug had been present on the Providers list too at lower visibility, and is now resolved on both pages.

Brand components are hand-mirrored from the financial dashboard for visual recognizability. Non-brand UI (tables, forms, dialogs, lists, detail panes) uses Tailwind utilities and shadcn/ui primitives freely with no dashboard-class-pattern constraint.
